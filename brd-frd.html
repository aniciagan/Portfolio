Business & Functional Requirements Documents (BRD & FRD)

Business Requirements Document (BRD)

Project Title: Singapore Traffic and COE Analysis Platform
Version: 1.2
Date: August 9, 2025

1. Introduction

1.1 Purpose

The purpose of this document is to define the business and technical requirements for a data analysis platform. This platform will be used to analyze the complex relationship between Certificate of Entitlement (COE) prices and traffic conditions across Singapore. The objective is to provide a comprehensive, data-driven tool for policymakers and urban planners to evaluate the effectiveness of current vehicle population control measures and inform future transportation strategies.

1.2 Scope

This project will deliver a web-based, analytical dashboard that ingests historical and real-time data from various sources, including the Land Transport Authority (LTA). The platform will provide a suite of tools for data visualization, correlation analysis, and predictive modeling to help stakeholders understand how fluctuations in COE prices, changes in the vehicle quota, and other factors impact traffic flow and congestion.

2. Goals and Objectives

  • Correlate Key Metrics: Analyze the historical relationship between COE prices, the vehicle population, and traffic congestion data to identify patterns and potential causal links.
  • Predictive Modeling: Develop and validate a model that can predict short-term changes in traffic congestion based on current COE price trends and other variables.
  • Enhance Data Transparency: Provide a centralized and interactive platform that makes complex transportation data accessible and easy to understand for stakeholders.
  • Inform Policy Decisions: Generate data-driven insights and recommendations to support the evaluation and refinement of Singapore's Vehicle Quota System (VQS).

3. Stakeholders

  • Project Sponsor: Ministry of Transport (MOT) leadership, responsible for funding and high-level strategy.
  • Product Owner: An urban planning or data analytics representative from the Land Transport Authority (LTA).
  • Data Science Team: The team responsible for building the data pipelines, analytical models, and the user-facing application.
  • Policymakers: LTA and MOT officials who will use the platform's insights for decision-making.
  • Academics and Researchers: External parties who may use the platform's data to conduct their own studies.

4. Assumptions

  • The project team will be granted full access to the required historical and real-time data feeds from the LTA DataMall APIs.
  • Necessary server infrastructure and computational resources will be allocated for data processing and model training.
  • The project's success is dependent on the quality and accuracy of the data provided by external sources.
  • The project will be developed following a phased approach, with an initial version focusing on data visualization before moving to more complex predictive modeling.

5. Functional Requirements

The platform must include the following features:

  • Data Ingestion Module: A system to automatically collect and parse data from LTA APIs, including `COE Bidding Results`, `Traffic Speed Bands`, `ERP Rates`, and other relevant traffic data.
  • Interactive Dashboard: A web-based dashboard with customizable data visualizations, including charts and graphs showing COE price trends, traffic speeds, and vehicle population over time.
  • Correlation Analysis Tool: A feature that allows users to select different variables (e.g., COE prices, time of day, weather) and instantly generate a correlation analysis.
  • Predictive Model Interface: An interface to input variables and run a pre-trained model to forecast potential traffic congestion scenarios.
  • Geospatial Visualization: An interactive map of Singapore that can overlay real-time traffic speeds and ERP gantry locations.
  • Reporting and Export: The ability to generate and export reports, graphs, and raw data in common formats (e.g., PDF, CSV).

6. Non-Functional Requirements

  • Performance: The platform must be able to process and render visualizations of large datasets in under 5 seconds. The predictive model must be able to return a forecast within 10 seconds of a user request.
  • Security: The application must comply with Singapore's government security standards, including secure user authentication and data encryption.
  • Usability: The user interface (UI) should be intuitive for non-technical users and responsive on both desktop and tablet devices.
  • Scalability: The architecture must be scalable to handle a potential increase in data volume from new LTA data sources (e.g., ERP 2.0) without requiring a complete redesign.
  • Reliability: The data ingestion pipelines must have robust error handling and be designed for high uptime.

7. Success Criteria

  • Predictive Accuracy: Achieve a minimum of 80% accuracy in the predictive model's ability to forecast traffic congestion in key areas.
  • Stakeholder Engagement: The platform is used by at least 15 LTA/MOT employees on a weekly basis within three months of launch.
  • Insight Generation: The platform is used to generate at least three new, actionable policy insights or recommendations within the first six months.
  • Positive Feedback: Achieve an average user satisfaction score of 4 out of 5 or higher in a post-implementation survey.

8. Constraints

  • Budget: The project must be completed within the allocated budget of $750,000.
  • Timeline: The initial version of the platform must be launched within 12 months.
  • Data Dependency: The project is highly dependent on the stability and accuracy of external LTA APIs. Any changes to these APIs will require updates to the platform.
  • Resources: The project team is limited to a maximum of five full-time employees.

9. User Stories

Here are some example user stories that capture the needs of the key stakeholders:

  • As a Policymaker, I want to see a historical trend of COE prices overlaid with average traffic speeds on major expressways, so that I can visually identify any correlation between them.
  • As a Data Scientist, I want to access raw data for COE bids and traffic speeds in a downloadable CSV format, so that I can perform custom statistical analysis and model validation.
  • As an Urban Planner, I want to use the geospatial visualization to see real-time traffic speeds across different districts, so that I can understand where congestion is most severe and when it occurs.
  • As a Project Sponsor, I want to view a dashboard with key performance indicators (KPIs) like the predictive model's accuracy and the number of generated reports, so that I can monitor the project's success.
  • As an Academic Researcher, I want to be able to run a predictive model with different hypothetical COE price scenarios, so that I can study the potential impact on future traffic conditions.

Functional Requirements Document (FRD)

This document details the specific functional requirements for the "Singapore Traffic and COE Analysis Platform", outlining the software's behavior and the expected user interactions.

1. Dashboard UI Requirements

  • Login Page: The platform shall present a secure login page for user authentication.
  • Dashboard Layout: The main dashboard shall feature a customizable, widget-based layout, allowing users to arrange and resize data visualization components.
  • Navigation: A sidebar or top navigation menu shall provide easy access to different sections, including "Dashboard," "Correlation Analysis," and "Predictive Modeling."
  • Date/Time Filters: All visualizations shall include dynamic filters for selecting specific date ranges and time periods (e.g., "Last 30 days," "Q2 2024," "Custom Range").

2. Data Visualization Requirements

  • Line Charts: The system shall display line charts for COE prices and traffic speeds over time. Users must be able to overlay multiple data series on a single chart for comparison.
  • Geospatial Map: An interactive map of Singapore shall be rendered. This map must be able to display traffic speed data as color-coded lines on major roads, with green for free-flowing traffic, yellow for moderate congestion, and red for heavy congestion. The map shall also pinpoint the locations of ERP gantries.
  • Pie Charts: The system shall generate pie charts to visualize the distribution of COE bids by category (e.g., Car, Motorcycle, Goods Vehicle).

3. Correlation Analysis Requirements

  • Variable Selection: Users shall be able to select at least two variables from a dropdown menu (e.g., `COE Price`, `Average Traffic Speed`, `Vehicle Population`) to run a correlation analysis.
  • Analysis Execution: Upon user request, the system shall calculate and display a correlation coefficient (e.g., Pearson correlation) between the selected variables.
  • Visualization: The results of the correlation analysis shall be visualized using a scatter plot with a regression line.

4. Predictive Modeling Requirements

  • Model Input: The system shall provide an interface where users can input hypothetical values for key variables (e.g., next COE price, new vehicle quota).
  • Forecast Generation: The system shall run the predictive model and generate a forecast of future traffic congestion, represented as a numerical value (e.g., average speed) and a confidence interval.
  • Scenario Comparison: Users shall be able to save and compare multiple predictive scenarios side-by-side.
